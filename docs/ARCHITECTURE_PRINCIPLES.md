## 🎯 核心设计哲学

### 静默成长 vs 游戏化体验

Anima-AIOS 采用**双层架构**，分离"成长引擎"与"体验呈现"：

```
┌─────────────────────────────────────────────────────────┐
│                   用户 (立文/日安)                       │
│              "让成长可见，让认知可量"                     │
└─────────────────────────────────────────────────────────┘
                          ↓ 对话交互
┌─────────────────────────────────────────────────────────┐
│         Anima-skill v5.x - 游戏化体验层                  │
│                                                          │
│  设计哲学：让成长过程有趣、可见、有仪式感                 │
│  迭代节奏：快速（1-2 周/版本）                            │
│  用户感知：强感知（像玩游戏一样有趣）                     │
│                                                          │
│  ┌────────────────────────────────────────────────────┐ │
│  │ 🎨 可视化：ASCII 卡片、进度条、雷达图、动画          │ │
│  │ 🎮 游戏化：任务、排行、成就、徽章、彩蛋            │ │
│  │ 💬 交互：对话式查询、即时反馈、情感化文案          │ │
│  │ 🎪 娱乐化：惊喜、仪式、个性化主题                   │ │
│  └────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
                          ↓ 调用（只读/写入数据）
┌─────────────────────────────────────────────────────────┐
│         Anima-core v5.x - 静默成长引擎                   │
│                                                          │
│  设计哲学：成长以静默无声的方式进行                       │
│  迭代节奏：稳定（1-2 月/版本）                            │
│  用户感知：无感知（像呼吸一样自然）                       │
│                                                          │
│  ┌────────────────────────────────────────────────────┐ │
│  │ 📊 数据：EXP 记录、维度分数、等级、历史             │ │
│  │ 🧮 算法：等级公式、归一化、质量评估、向量检索      │ │
│  │ 🛡️ 保护：备份、权限、安全、加密                    │ │
│  │ 📐 规则：EXP 获取规则、任务生成逻辑、去重算法       │ │
│  └────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
                          ↓ 存储
┌─────────────────────────────────────────────────────────┐
│                  文件系统 (~/.anima/)                    │
│  exp_history.jsonl, cognitive_profile.json, ...         │
└─────────────────────────────────────────────────────────┘
```

## 📐 关注点分离（Separation of Concerns）

| 维度 | Anima-core | Anima-skill |
|------|------------|-------------|
| **职责** | What（是什么） | How（怎么呈现） |
| **变更频率** | 低（稳定） | 高（灵活） |
| **用户感知** | 无感知 | 强感知 |
| **测试重点** | 正确性、安全性、性能 | 体验、交互、视觉 |
| **回滚成本** | 高（数据风险） | 低（展示问题） |
| **团队分工** | 后端/算法/数据 | 前端/UX/交互 |
| **版本策略** | 向后兼容优先 | 用户体验优先 |
| **迭代驱动** | 问题驱动（bug/性能） | 用户驱动（反馈/需求） |

## ✅ 正确的迭代路径

### 场景 1：增强游戏化体验 → 迭代 skill

**需求示例：**
- "升级时想要更炫酷的动画"
- "想要收集成就徽章"
- "排行榜想要显示变化趋势箭头"
- "想要个性化卡片主题"

**✅ 正确做法：迭代 skill**
```python
# 在 skill 中添加升级动画效果
def celebrate_level_up(old_level, new_level):
    animation = generate_celebration_animation()
    card = generate_fancy_card()
    message = generate_emotional_message()
    return {"animation": animation, "card": card, "message": message}
```

**❌ 错误做法：修改 core**
```python
# 不应该在 core 中添加 UI 逻辑
# core 不应该关心"动画"、"卡片样式"等展示细节
```

---

### 场景 2：优化业务规则 → 迭代 core

**需求示例：**
- "semantic 记忆应该获得更多 EXP"
- "每日 EXP 上限需要调整"
- "等级公式需要优化"
- "去重算法需要改进"

**✅ 正确做法：迭代 core**
```python
# 在 core 中修改 EXP 规则
class EXPTracker:
    EXP_RULES = {
        'episodic': 1,
        'semantic': 2,  # 从 1 改为 2
        'weekly_report': 5
    }
```

**❌ 错误做法：在 skill 中硬编码**
```python
# 不应该在 skill 中修改业务规则
# skill 不应该决定"semantic 应该获得多少 EXP"
```

- "想要等级趋势折线图"

**✅ 正确做法：迭代 skill**
```python
# 在 skill 中实现可视化
def generate_radar_chart(dimensions):
    # skill 负责 ASCII 艺术生成
    # core 只提供原始数据
    return ascii_radar
```

## 🔄 数据流向规范

### 单向依赖

```
skill → core → 数据
```

- ✅ skill 可以调用 core
- ❌ core 不应该依赖 skill
- ❌ skill 和 core 不应该循环依赖

### 接口规范

**core 返回结构化数据：**
```python
# ✅ 正确
return {
    "level": 10,
    "exp": 5238,
    "next_level_exp": 5239,
    "dimensions": {
        "internalization": 85,
        "application": 78,
        ...
    }
}

# ❌ 错误（core 不应该生成 UI 字符串）
return "╔════════════════════════╗\n║ Lv.10 ║\n╚════════════════════════╝"
```

**skill 负责包装为体验：**
```python
# ✅ 正确（skill 添加可视化）
profile = core.get_cognitive_profile()
card = f"""
╔══════════════════════════════════════════╗
║         {profile['agent']} - 认知画像 Lv.{profile['level']}            ║
╠══════════════════════════════════════════╣
║  EXP: {profile['exp']} / {profile['next_level_exp']}  {generate_bar(profile['progress'])}        ║
╚══════════════════════════════════════════╝
"""
return {"card": card, "data": profile}
```

## 🚫 禁止事项

### core 层禁止：
- ❌ 生成 UI 字符串（ASCII、HTML、Markdown 表格）
- ❌ 处理用户交互逻辑（对话、情感化文案）
- ❌ 硬编码展示逻辑（卡片样式、颜色）
- ❌ 依赖 skill 层的任何模块
- ❌ 频繁变更 EXP 规则（影响数据一致性）

### skill 层禁止：
- ❌ 修改核心业务规则（EXP 计算、等级公式）
- ❌ 绕过 core 直接读写数据文件
- ❌ 在 skill 中实现去重、限额等核心逻辑
- ❌ 假设 core 的返回格式（应使用类型检查）

## 📊 决策树

当接到新需求时，使用以下决策树判断迭代哪一层：

```
新需求
  │
  ├─ 是否涉及 UI/可视化/展示？
  │   ├─ 是 → 迭代 skill ✅
  │   └─ 否 → 继续判断
  │
  ├─ 是否涉及业务规则（EXP、等级、维度）？
  │   ├─ 是 → 迭代 core ✅
  │   └─ 否 → 继续判断
  │
  ├─ 是否涉及数据读写/存储？
  │   ├─ 是 → 迭代 core ✅
  │   └─ 否 → 继续判断
  │
  ├─ 是否涉及用户交互/对话？
  │   ├─ 是 → 迭代 skill ✅
  │   └─ 否 → 继续判断
  │
  └─ 是否涉及性能优化？
      ├─ 算法性能 → 迭代 core ✅
      └─ 展示性能 → 迭代 skill ✅
```
